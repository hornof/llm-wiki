### 4-condition canonical Loop-Engineering canonical-pre-build canonical-test

> *"Loops earn their cost under four conditions. Miss one and the loop costs more than it returns."*

| Condition | Canonical-articulation |
|---|---|
| **1. The task repeats** | *"A loop amortizes its setup across many runs. For a one-time job, a good prompt is faster and cheaper."* |
| **2. Verification is automated** | *"A test suite, a type checker, a linter, a build. No automated check means you're back in the chair reading every diff — the exact job the loop was supposed to remove."* |
| **3. Your token budget can absorb the waste** | *"Loops re-read context, retry, explore. That burns tokens whether or not the run ships anything. The technique scales with budget."* |
| **4. The agent has a senior engineer's tools** | *"Logs, a reproduction environment, the ability to run the code it writes and see what breaks. Without that, the loop iterates blind."* |

**First wiki-captured 4-condition canonical Loop-Engineering canonical-pre-build canonical-test**. **First wiki-captured "miss one and the loop costs more than it returns" canonical-discipline framing**.

### "Who wins, who loses" canonical-economics framing

> *"The economics are not universal. The people calling loop engineering obvious tend to have unmetered tokens. The people for whom it's reckless are usually on a $20 consumer plan trying to run heavy verification loops without hitting limits or a surprise invoice."*

**Who wins**:
- Teams with repetitive, machine-checkable work + budget
- Codebases with strong existing test suites
- Async-first teams with multi-agent patterns

**Who loses**:
- Solo builders on consumer plans (*"the token bill arrives before the productivity gain does"*)
- Anyone with no automated verification (*"a loop with no real check is the agent agreeing with itself on repeat"*)
- Teams whose real constraint is review capacity (*"a loop generates more code; if review was already the bottleneck, it just makes the queue longer"*)
